1. 성능 (Performance)

P1 높음 N+1 쿼리 문제
data_service.py:541~697
extract_and_save_cash_flows_from_data_record() 내 루프에서 행마다 SELECT → commit → refresh 실행. 50개 행 처리 시 150+ 쿼리 발생.
수정 방향: 루프 종료 후 bulk upsert + 단일 commit으로 전환
P2 높음 Excel 파일 이중 로딩
excel_parser.py:60~65
data_only=True와 data_only=False로 동일 파일을 두 번 메모리에 로드. 대용량 파일 시 메모리 2배 사용.
수정 방향: data_only=True 단독 처리 후, None 셀 발견 시에만 formula 워크북 lazy 로딩
P3 중간 스케줄러 풀스캔
scheduler_service.py:26, 86
30초마다 db.query(SheetData).all() 전체 조회. 데이터 증가에 따라 선형 성능 저하.
수정 방향: 마지막 sync 이후 updated_at이 변경된 시트만 처리
P4 낮음 렌더링마다 함수 재생성
DataVisualization.tsx:17
formatNumber가 컴포넌트 내부에 선언되어 매 렌더링마다 재생성.
수정 방향: src/utils/format.ts로 분리

2. 가독성 (Readability)

R1 높음 거대 함수 (365줄)
data_service.py:334~699
extract_and_save_cash_flows_from_data_record()가 단독 365줄. 중첩 if-else 5단계 이상.
수정 방향: 내부 헬퍼 4개로 분리
_find_section_header() · _find_column_headers() · _extract_row_data() · _merge_monthly_data()
R2 중간 동일 패턴 5회 중복
data_service.py:373~414 외
isinstance(raw, (int, float)) → str 변환 블록이 함수 내에서 5회 이상 반복.
수정 방향: _to_str(value) 헬퍼 함수로 추출
R3 낮음 함수 내부 import
data_service.py:455
import re가 함수 내부에 위치.
수정 방향: 파일 최상단으로 이동
R4 중간 DRY 위반 — 중복 유틸 함수
excel_parser.py:10~27 / data_service.py:91~108
convert_datetime_to_string() 동일 로직이 두 파일에 독립적으로 존재.
수정 방향: data_service.py의 private 함수 제거, excel_parser의 공개 함수 import로 대체
R5 중간 하드코딩된 매직값
data_service.py:405, 437, 529~533
컬럼 인덱스 "1"~"5", range(4, 17), 긴 키워드 리스트가 로직 내부에 산재.
수정 방향: 모듈 상단 상수로 선언 (COLUMN_ITEM_NAME = "1" 등)

4. 보안 (Security)

S1 높음 파일명 미검증
upload.py:28
file.filename이 None일 경우 AttributeError 발생. Path Traversal 공격 미방어 (../ 포함 가능). MIME 타입 검증 없음.
수정 방향: 
if not file.filename:
    raise HTTPException(status_code=400, detail="파일명이 없습니다.")
safe_name = os.path.basename(file.filename)  # Path traversal 방어
S2 높음 파일 크기 제한 없음
upload.py 전체
타임아웃(10분)만 있고 용량 상한이 없어 서버 메모리 고갈 위험.
수정 방향: 업로드 크기 상한 설정 (예: 50MB)
S3 중간 CORS 과도 허용
main.py:36~37
allow_methods=["*"], allow_headers=["*"]로 모든 메서드·헤더 허용.
수정 방향: allow_methods=["GET", "POST"]로 제한

리팩터링 실행 계획

1 Phase 1 — 즉시 수정 (Low Risk · Zero Breaking Change)
  • S1 file.filename None 체크 + os.path.basename() sanitize 추가
  • S2 파일 크기 상한 설정 (50MB)
  • S3 CORS allow_methods=["GET", "POST"]로 축소
  • R3 import re 파일 최상단으로 이동
  • R4 data_service._convert_datetime_to_string() 제거 → excel_parser import로 대체
  • M4 formatNumber → frontend/src/utils/format.ts 분리
  • FastAPI on_event("shutdown") → lifespan 패턴으로 변경
2 Phase 2 — 구조 개선 (Medium Risk · Non-Breaking)
  • R2 _to_str() 헬퍼 추출 후 중복 5회 일괄 교체
  • R5 매직 상수 모듈 상단으로 추출 (COLUMN_*, EXCLUDE_KEYWORDS 등)
  • M2 data.py 라우터의 직접 DB 쿼리 → data_service 함수로 위임
3 Phase 3 — 모듈 분리 (Medium-High Risk · Requires Regression Check)
  • M1 data_service.py → date_utils.py / crud.py / customer_service.py / cash_flow_service.py 4개 파일로 분리
  • R1 거대 함수 내부 헬퍼 4개 추출 (_find_section_header 등)
  • M3 upload.py 로직 → upload_service.py 분리
4 Phase 4 — 성능 개선 (High Complexity)
  • P1 현금흐름 upsert → bulk INSERT/UPDATE + 단일 commit으로 전환
  • P2 Excel 이중 로딩 → lazy 로딩 방식으로 개선
  • P3 스케줄러 → 변경된 시트만 처리하도록 개선
5 Phase 5 — 테스트 추가 (After Refactoring)
  • T1 backend/tests/ 디렉토리 + pytest 설정
  • T1 data_service 핵심 추출 로직 단위 테스트
  • T2 API 엔드포인트 통합 테스트
